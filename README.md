# Мой GitLab дома: без облачного GitLab и без публикации секретов

Это короткий публичный конспект о том, как поднять свой GitLab у себя дома и открыть к нему внешний доступ без публикации паролей, приватных IP, SSH-ключей и прочих чувствительных данных.

## Зачем это вообще

Идея простая: GitLab работает на домашнем mini-PC, а снаружи открывается по обычной ссылке в браузере.

В моем случае итоговый публичный адрес:

- [https://gitlub.ru](https://gitlub.ru)

Проект для теста:

- [https://gitlub.ru/MEGA/deveps-mega-coder](https://gitlub.ru/MEGA/deveps-mega-coder)

Если регистрация на инстансе включена, можно подать заявку на аккаунт. Новый пользователь может требовать ручного подтверждения администратором.

## Почему `gitlub.ru`, а не `gitlab.ru`

Потому что `gitlab.ru` на момент запуска уже был занят, и для собственного self-hosted экземпляра понадобился другой короткий домен.

`gitlub.ru` выбран как отдельное, понятное и свободное имя для собственного инстанса.

## Почему не просто `gitlab.com`

Этот репозиторий не про GitLab.com, а про self-hosted GitLab.

Причина простая: self-hosted вариант дает полный контроль над:

- доменом
- доступом
- регистрацией пользователей
- резервным копированием
- обновлениями
- правилами входа

Для части пользователей GitLab.com может оказаться неудобным в онбординге. В официальной документации GitLab есть отдельная тема про identity verification на GitLab.com:

- всем пользователям может потребоваться email verification
- части пользователей может потребоваться phone verification
- для high-risk сценариев может потребоваться credit card verification

При этом в официальной документации GitLab для phone verification указано, что для `Russia` поддержка помечена как `Unsupported`.

Поэтому в таком сценарии self-hosted GitLab — это не "экзотика", а нормальный способ убрать лишние зависимости и самому контролировать доступ.

## Как устроена схема

Безопасная идея такая:

1. GitLab работает локально на домашнем mini-PC.
2. Наружу не публикуется сам домашний роутер и не пробрасываются лишние порты.
3. Отдельный VPS принимает входящий трафик по домену.
4. VPS проксирует запросы до домашнего GitLab через постоянный обратный туннель.
5. HTTPS выпускается на VPS через Let's Encrypt.

В итоге:

- данные и сам GitLab остаются дома
- снаружи люди открывают обычный домен
- домашний сервер не раскрывает лишнюю информацию

## Что понадобилось

- mini-PC с Linux
- Docker Compose
- GitLab CE
- домен
- VPS с белым IP
- Nginx или Caddy на VPS
- reverse tunnel от домашнего сервера к VPS

## Что получилось на практике

- веб-вход работает по [https://gitlub.ru](https://gitlub.ru)
- страница логина открывается по [https://gitlub.ru/users/sign_in](https://gitlub.ru/users/sign_in)
- проект открывается по адресу вида `https://gitlub.ru/<group>/<project>`

## Почему это удобно

- не зависит от готового SaaS-аккаунта
- можно использовать свой домен
- можно включать или выключать регистрацию
- можно подтверждать новых пользователей вручную
- можно держать репозитории у себя

## Регистрация пользователей

Для публичного self-hosted GitLab лучше сразу проверить ограничения на регистрацию.

Полезные настройки:

- включена ли регистрация вообще
- требуется ли ручное подтверждение администратора
- требуется ли подтверждение email
- есть ли ограничения по доменам email

Официальная документация GitLab:

- [Sign-up restrictions](https://docs.gitlab.com/administration/settings/sign_up_restrictions/)
- [Make new users confirm email](https://docs.gitlab.com/security/user_email_confirmation/)
- [Moderate users](https://docs.gitlab.com/administration/moderate_users/)

## Можно ли присылать админу письмо о новой регистрации

Коротко: ручное подтверждение новых пользователей включить можно.

Что есть из коробки по официальной документации:

- пользователь может попасть в состояние `Pending approval`
- администратор может зайти в `Admin -> Overview -> Users`
- там можно одобрить или отклонить регистрацию

Что важно:

- в официальной документации GitLab я не нашел отдельной штатной настройки вида "отправлять администратору письмо при каждой новой заявке на регистрацию"

То есть базовый вариант такой:

- включить `Require admin approval for new sign-ups`
- включить нормальную почту для самого GitLab
- периодически проверять пользователей в состоянии `Pending approval`

Если нужно именно письмо админу о каждой новой заявке, это обычно добирается отдельно:

- через внешний мониторинг
- через скрипт/cron
- через API
- через дополнительную автоматизацию поверх GitLab

## Почта в своем GitLab

Чтобы работали исходящие письма, нужно отдельно настроить SMTP.

Официальная документация:

- [Omnibus GitLab SMTP settings](https://docs.gitlab.com/omnibus/settings/smtp/)

## Что важно не публиковать

Если вы делаете публичный репозиторий с таким кейсом, не выкладывайте:

- реальные пароли
- приватные IP
- приватные SSH-ключи
- токены
- содержимое `.env`
- дампы конфигов с секретами

Публиковать безопасно:

- общую схему
- список компонентов
- команды без секретов
- скриншоты интерфейса без чувствительных данных

## Минимальный план для тех, кто хочет повторить

1. Поднять GitLab CE у себя дома.
2. Убедиться, что локально он работает.
3. Купить или подключить свой домен.
4. Поднять простой VPS.
5. Настроить обратный туннель от дома к VPS.
6. Настроить Nginx на VPS.
7. Выпустить HTTPS-сертификат.
8. Проверить регистрацию, email и ограничения на sign-up.

## Для теста

Если на инстансе включена регистрация и администратор подтверждает новые аккаунты вручную, можно протестировать вход через:

- [https://gitlub.ru/users/sign_in](https://gitlub.ru/users/sign_in)
